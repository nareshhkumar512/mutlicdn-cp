## 🔄 REQUEST FLOW (Data Path)

```
User creates XDeliveryService claim
         ↓
Crossplane Composition Engine (xdeliveryservice-demo)
         ↓
Patch & Transform Function renders templates
         ↓
Transforms input into adapter configurations:
    ├─ Summary: metadata snapshot
    ├─ Certificate: TLS provisioning request
    ├─ DNS: Internal routing (internal-api mode)
    ├─ Identity: IDAnywhere credentials
    ├─ Cloudflare: Native API schema
    └─ Akamai: Terraform variables
         ↓
All ConfigMaps created in Kubernetes
         ↓
External adapters read ConfigMaps:
    ├─ Akamai adapter executes Terraform module
    ├─ Cloudflare adapter calls API
    ├─ Certificate adapter provisions TLS
    ├─ DNS adapter routes traffic
    └─ Identity adapter configures auth
         ↓
Service READY when all adapters succeed
```

## 🔧 TROUBLESHOOTING MATRIX

| Symptom | Root Cause | Fix |
|---------|-----------|-----|
| ConfigMap unmarshal error | Nested objects in data field (e.g., `tfvars.json`) | Use flat key names only (`tfvars`) |
| SYNCED=False | Composition error, missing provider, or manifest syntax | Check logs: `kubectl logs -n crossplane-system` |
| READY=False after 5min | Adapters not executing or erroring | Inspect ConfigMaps, verify adapter credentials |
| Cannot create XDeliveryService | XRD not installed | Run `02-install-demo.sh` |
| nginx-service has no IP | LoadBalancer pending | Wait for AWS to provision, check `kubectl get svc -n default` |

## 🎨 COMPOSITION RENDERING LOGIC

### Patch Types Used
1. **FromCompositeFieldPath** — Copy single field values
   ```yaml
   - type: FromCompositeFieldPath
     fromFieldPath: spec.serviceName
     toFieldPath: spec.forProvider.manifest.metadata.name
     transforms:
       - type: string
         string:
           fmt: "%s-akamai-terraform-request"  # Adds suffix
   ```

2. **CombineFromComposite** — Merge multiple values into JSON
   ```yaml
   - type: CombineFromComposite
     combine:
       strategy: string
       string:
         fmt: '{"service_name":"%s","hostname":"%s",...}'
       variables:
         - fromFieldPath: spec.serviceName
         - fromFieldPath: spec.hostname
     toFieldPath: spec.forProvider.manifest.data.tfvars
   ```

## 💡 KEY DESIGN PATTERNS

1. **Composite Resource Pattern** — XDeliveryService abstracts complexity
2. **Pipeline Composition** — Multi-step orchestration with patches
3. **ConfigMap as Contract** — Adapters consume ConfigMap data
4. **Adapter Pattern** — Pluggable external system integrators
5. **Declarative IaC** — GitOps-friendly YAML manifests
6. **Status-Driven Reconciliation** — Continuous alignment with desired state
